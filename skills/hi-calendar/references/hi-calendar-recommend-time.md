# hi-calendar-recommend-time

用于生成或解释时间候选。时间推荐的目标是帮助用户选择方案，不是直接创建会议。

## 两类时间候选

### 1. 一体化候选：时间 + 会议室

使用 `calendar:get-intelligent-recommendation`。适用条件：

- 单次会议；
- 已有开始时间、结束时间、参会人；
- 用户没有明确表示不需要会议室；
- 用户要创建会议，或想要“时间和会议室一起给方案”。

这是一体化候选生成器，也是会议推荐的重点路径。它不要求先查共同空闲，也不要先追问具体会议室或区域；用户只给一个对方参会人时可以直接调用。输出时说明 `timeEnough`、`meetingEnough` 和候选会议室。不要把它降级成“先手动查人忙闲，再手动查会议室”的组合流程，除非用户明确只要纯时间、明确无会议室，或需要解释具体冲突原因。

### 2. 纯时间候选：共同空闲 / 时间解释

使用 `calendar:get-common-free-time` 或 `calendar:get-user-schedules`。适用条件：

- 用户明确只想看共同空闲；
- 用户明确不需要会议室 / 线上会议；
- 用户要解释“为什么这个时间不合适 / 谁冲突”；
- 智能推荐不适用，例如循环会议、只做可用性分析。

纯时间结果如果后续要创建会议，必须回到 `hi-calendar-writing.md` 做最终确认；若仍需要会议室，再由写入流程决定是否进入智能推荐或纯会议室分支。

## 推荐策略

1. 相对时间、自然语言时间、跨时区场景，先读 `hi-calendar-checks-and-timezone.md` 解析基准时区。
2. 单次会议且需要会议室时优先一体化候选；纯时间候选只在用户明确只看共同空闲、明确无会议室或智能推荐不适用时使用。
3. 推荐时优先选择所有参会人都处在推荐时段内的候选。
4. 没有完美候选时，可以给少量谨慎选项，但必须说明影响对象、本地时间和为什么属于折中。
5. 用户给出不可用区间时，优先通过 `get-common-free-time --exclude` 传入绝对时间区间；相对规则先展开成绝对区间。
6. 需要解释冲突原因时，用 `calendar:get-user-schedules` 拉取忙碌事实；需要完整事实时按 `hasMore` 翻页。

## 输出要求

候选时间至少展示：

- 时间段；
- 是否满足共同空闲；
- 跨时区时各关键参会人的本地时间；
- 是否触及非推荐时段；
- 下一步：选择候选、调整范围、是否继续创建。

不要把“推荐候选”写成“已安排”。用户选定候选后，转 `hi-calendar-writing.md`。

## 注意事项

- 用户明确不需要会议室时，不要为了补会议室再回到智能推荐。
- 只有创建人、没有其他参会人时，不要调用智能推荐；纯时间分支不受“创建人 + 其他参会人”限制。
- 用户明确要求跳过推荐 / 检查时，可以进入写入流程，但要说明风险。
- 推荐时段、时区、分页和冲突解释遵循 `hi-calendar-checks-and-timezone.md`。
