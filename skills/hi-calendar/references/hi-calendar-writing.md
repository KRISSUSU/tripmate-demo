# hi-calendar-writing

用于处理普通会议创建、循环会议创建、编辑会议。写操作的核心不是凑齐命令参数，而是把时间、人员、空间、材料和风险收敛成用户确认过的执行计划。

## 写操作状态机

1. **分类**：单次创建、循环创建、编辑已有日程。
2. **收敛事实**：时间、人、空间、文档、循环范围、时区、权限。
3. **生成候选**：按用户目标使用智能推荐、纯时间、纯会议室或详情查询。
4. **确认计划**：复述最终字段、候选来源、风险和即将执行的命令。
5. **执行写入**：确认后生成 operateCode（如需要），再调用写命令。

任何推荐、查询、冲突检查都不是写入成功；用户确认前不得调用 create / create-recurring / edit-schedule。

## 资源模型

- **时间**：开始时间、结束时间 / 时长、是否循环、用户表达时间的基准时区。
- **人员**：参会人账号；自然语言姓名先用 `search:employee --query` 解析；同名候选必须确认；涉及跨时区时要能解释本地时间。
- **空间**：会议室、线上/无会议室、区域/容量等筛选约束、候选来源。
- **材料**：是否关联文档、文档 URL / shortcutId、是否赋权或创建任务。
- **授权**：用户是否明确确认最终执行计划。

文档是标准要素。创建会议时即使用户没有主动提到，也要在最终确认前明确“无文档”或具体文档方案。

## 单次会议创建

1. 先补齐智能推荐所需的最小事实：开始时间、结束时间、参会人。若缺少这些事实，先询问或解析，不要调用推荐命令；参会人是姓名、薯名等自然语言时，先用 `search:employee --query` 解析到账号。
2. 若用户明确表示“不需要会议室 / 线上会议 / 无需会议室”，不要用智能推荐补会议室；按无会议室创建继续。需要验证参会人时间时，用 `calendar:get-common-free-time` 或忙碌事实检查。
3. 若用户没有排除会议室，且开始时间、结束时间、参会人已明确，默认调用 `calendar:get-intelligent-recommendation`。这是单次会议的重点推荐路径，不要退回手动查人忙闲 + 查会议室，也不要在调用前强行追问具体会议室或区域。
4. 展示智能推荐候选，**必须让用户选择确认**：
   - 智能推荐返回的是“时间 + 会议室”组合候选；**必须把候选时间和候选会议室一起展示给用户选择**，不要只展示时间或只展示会议室。
   - `timeEnough=true` 表示该候选时间对参会人可行。
   - `meetingEnough=true` 且 `meetingList` 非空时，会议室是候选资源，需要用户选择。
   - 时间或会议室不足时，说明限制，并让用户选择调整时间、只看共同空闲、保留时间另找会议室，或按风险继续。
5. 用户选定候选后，回到写前确认；不要在推荐分支里直接创建。
6. 最终确认前补齐标题/内容、文档决策、腾讯会议偏好等写入字段。
7. 用户确认后，调用 `utils:generate-operate-code` 获取操作码，再调用 `calendar:create`。失败重试复用同一个 operateCode，最多 3 次。

### 无会议室 / 线上 / 普通占位

- 用户明确不需要会议室时，最终计划中写明“会议室：无 / 线上”。
- 若只是创建一个普通日程占位、无参会人、无会议室，确认这是普通占位日程后可直接进入写前确认。
- 若用户表达的是“别让人约我 / 屏蔽这段时间 / 专注 / 勿扰”，优先转 `hi-calendar-focus-time.md`，不要用普通会议创建替代专注时间。
- 若用户要求跳过忙闲 / 时区检查，允许继续，但要在确认计划中写明跳过项和风险。

## 循环会议创建

`calendar:get-intelligent-recommendation` 只适合单次会议，不用于循环会议。

1. 确认重复规则：频率、星期几、间隔、系列结束时间。
2. 确认参会人、是否需要会议室、文档决策。
3. 对首个目标时段做必要的参会人/会议室风险检查；若用户要求跳过，说明风险。
4. 写前确认必须包含循环范围和影响：首个时间、重复规则、结束时间、参会人、会议室、文档。
5. 用户确认后生成 operateCode，再调用 `calendar:create-recurring`。

## 编辑已有会议

编辑不是直接改字段，先定位并理解当前状态。

1. 先用 `calendar:get-user-schedules` 或 `calendar:get-schedule-detail` 定位目标；多个候选时让用户选择。
2. 读取详情，展示当前时间、参会人、会议室、文档、循环信息。
3. 确认变更差异：时间、参会人、会议室、文档、腾讯会议、循环范围；新增参会人是姓名或薯名时，先用 `search:employee --query` 解析并让用户确认。
4. 若变更涉及时间、参会人或会议室，按本文件的资源模型重新收敛事实和候选；必要时转推荐时间或会议室分支。
5. 写前确认展示“旧值 → 新值”和影响范围；用户确认后调用 `calendar:edit-schedule`。

## 候选分支的回流

- **智能推荐 → 写入**：用户选定时间和会议室后，回到最终确认。
- **纯时间 → 写入**：用户选定时间后，若仍需要会议室且是单次会议，回到智能推荐收敛会议室；若明确无会议室，直接进入最终确认。
- **纯会议室 → 写入**：用户选定会议室后，回到最终确认。
- **冲突解释 → 写入**：解释只提供事实；若用户调整方案，重新收敛候选；若用户 override，确认风险后写入。

## 写前确认模板

写前确认应包含以下字段，缺项要说明原因或继续补齐：

- 操作：创建 / 创建循环 / 编辑。
- 时间：开始、结束、时区；跨时区时展示关键参会人的本地时间。
- 参会人：账号或可读名称；同名确认结果。
- 会议室：具体会议室、线上/无会议室，或用户确认跳过。
- 文档：文档链接/shortcutId/授权设置，或“无文档”。
- 风险：冲突、非推荐时段、跳过检查、权限限制。
- 即将执行的命令：例如 `calendar:create` / `calendar:create-recurring` / `calendar:edit-schedule`。

只有用户明确确认后才能执行写命令。

## 幂等

- 普通创建和循环创建需要 operateCode，真正写入前调用 `utils:generate-operate-code`。
- 不同写操作不能共用 operateCode。
- 同一操作失败重试时复用同一个 operateCode，最多 3 次。

## 最小必要查询

- 有明确 `scheduleId` 或已定位目标时，直接读取详情，不重新全局搜索。
- 用户给定时间范围时，不主动扩大范围；需要扩大时先说明原因并征得同意。
- 查询失败时说明失败接口和影响，不要把失败当成没有冲突或没有结果。
