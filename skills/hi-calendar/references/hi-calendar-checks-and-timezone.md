# hi-calendar-checks-and-timezone

用于获取事实和解释风险：时区、推荐时段、忙闲冲突、会议室占用、分页和 override 风险。

## 事实命令定位

| 命令                            | 作用                                           |
| ------------------------------- | ---------------------------------------------- |
| `calendar:get-timezone`         | 查询创建人和参会人的 IANA 时区、本地时间       |
| `calendar:get-user-schedules`   | 查询忙碌事实；解释冲突、查日程、统计时需要翻页 |
| `calendar:get-common-free-time` | 快速验证共同空闲或生成纯时间候选               |
| `calendar:get-room-schedules`   | 检查会议室固定时段占用                         |

智能推荐是候选生成器；本 reference 是事实解释层。若智能推荐结果足以让用户选择，不要重复查共同空闲；只有需要解释“谁忙 / 为什么不合适 / 证据是什么”时，再查忙碌事实。

## 时间与时区

- 所有相对时间表达（今天、明天、本周、最近）都基于当前时间解释。
- 用户表达自然语言时间、相对时间、跨时区或多人会议时， 可同时查看创建人和参会人是否在同一个时区。
- 有参会人时，用 `calendar:get-timezone` 一次性查询创建人 + 所有参会人；创建人即使不在参会人列表中也要包含。
- 创建人时区缺失时停止询问，不要静默假设。
- 参会人时区缺失时可按创建人时区继续，但必须提示风险。
- 完整 ISO 时间已经包含绝对时间信息，不要再按其他时区重新解释或二次换算。
- 展示时区时使用用户可读名称（如北京时间、美东时间），不要只写 offset。

## 推荐时段策略

默认推荐时段按参会人本地时间判断：

- `09:00-12:00`
- `14:00-18:00`
- `19:00-21:00`

忙碌日程是硬冲突；非推荐时段是软约束。没有完美候选时可以给谨慎选项，但要说明影响对象和本地时间。

## 忙闲与冲突解释

- 需要解释参会人冲突时，用 `calendar:get-user-schedules` 获取忙碌事实。
- 目标时间必须被共同空闲完整覆盖，才可说所有人该时段都有空。
- `scheduleList` 是忙碌事实，不是最终空闲结果；推导空闲时要从查询窗口扣减忙碌区间。
- 需要完整判断、统计或共同空闲分析时，若 `hasMore=true` 必须翻页直到结束。
- 只做浏览展示时可先展示首页，并询问是否继续查看更多。
- 会议室有任何与目标时间重叠的预约都视为冲突。

## override 规则

用户明确说“直接创建 / 直接修改 / 别查空闲 / 忽略冲突 / 跳过时区”时，可以跳过对应检查，但必须满足：

1. 写入必要参数已经齐全。
2. 不跳过真正需要用户决策的事项，如缺失时间、同名人员、会议室候选、文档、循环范围。
3. 写前确认中明确说明跳过了哪些检查和可能风险。

## 失败处理

- 查询失败时说明失败接口、关键参数和影响。
- 不要把接口失败伪装成无结果。
- 不要为了完成任务而换日期、换目标、扩大范围或使用旧缓存。
