# hi-calendar-output-guidelines

用于统一候选、待确认和执行结果的表达，避免把查询或推荐说成已经写入。

## 输出骨架

默认按以下顺序输出：

1. **结论**：当前状态是什么，例如“找到 3 个候选”“还缺会议时长”“待你确认后创建”。
2. **详情**：候选、冲突、差异、限制或执行结果。
3. **下一步**：用户需要选择、补充、确认，或可继续执行什么。

## 状态用语

- **候选态**：使用“候选”“可选方案”“建议方案”，不要说“已安排”。
- **待确认态**：使用“我将按以下计划执行，确认后再创建/修改/取消”。
- **已执行态**：只有写命令成功后，才使用“已创建”“已修改”“已取消”。
- **失败态**：说明失败接口、关键参数、错误码/消息和影响。

## 表格使用

多条同结构记录、候选比较、多时区映射、冲突列表优先用表格。字段优先保持稳定：

- 对象：人、会议室、日程、候选项。
- 时间：时间段、本地时间、可用窗口。
- 状态：可用、冲突、待确认、受限、未知。
- 说明：限制原因、风险、权限、备注。
- 下一步：选择、确认、调整。

信息很少时用列表或短段落即可，不要强行表格。

## 候选展示

智能推荐结果是“时间候选 + 会议室候选”的组合，是会议推荐的重点输出形态。展示时**必须保留**两层信息：先列候选时间，再列该时间下可选会议室。若同一时间下有多个 `meetingList`，不要静默挑选会议室；让用户按候选编号选择时间和房间。

候选结果至少说明：

- 候选编号，便于用户选择。
- 时间段；跨时区时展示关键参会人的本地时间。
- 会议室名称/区域/容量（如果有）。
- 可用性或限制：`timeEnough`、`meetingEnough`、冲突、非推荐时段。
- 下一步：选择某个候选、调整条件，或确认继续。

会议纪要定位场景下，候选展示应最小披露：

- 展示编号、标题、时间、创建人/主持人展示名、是否有可用 `meetingId`、纪要查询状态。
- 不展示 `meetingCode`、`txMeetingCode`、`joinUrl`、完整参会人账号列表、文档链接、头像 URL 或原始 payload。
- `meetingId` 默认作为内部字段使用；只有用户明确需要排查时才脱敏展示。
- 若候选没有 `tencentMeetingDetail.meetingId`，只说明“该候选缺少可查询纪要所需 meetingId”，不要把会议号或链接暴露出来让用户复制。

示例结构：

| 候选 | 时间 | 会议室选项 | 状态 | 下一步 |
| --- | --- | --- | --- | --- |
| A | 14:00-15:00 | A1 / A2 | 时间可行，会议室可选 | 选择 A1 或 A2 |
| B | 15:00-16:00 | 无可用会议室 | 时间可行，会议室不足 | 换时间或另找会议室 |

## 写前确认展示

写操作前必须结构化复述：

| 字段   | 内容                                     |
| ------ | ---------------------------------------- |
| 操作   | 创建 / 创建循环 / 编辑 / 取消            |
| 时间   | 开始、结束、时区；跨时区本地时间         |
| 参会人 | 名称或账号                               |
| 会议室 | 具体房间、线上/无会议室、或跳过说明      |
| 文档   | 文档链接/授权，或无文档                  |
| 风险   | 冲突、非推荐时段、跳过检查、权限限制 |
| 下一步 | 等待用户确认后执行的命令                 |

用户未确认前，不要调用写命令。

## 冲突和无结果

- 先说限制本身，再说影响对象，最后给替代动作。
- 无结果时说明查询范围和约束，不要简单说“没有”。
- 权限不足时标注可见信息范围，不要补造缺失详情。
- 查询失败时不要继续写入，除非用户明确 override 且必要事实齐全。

## 时间表达

- 同一轮输出中时间格式保持一致。
- 有时区时使用可读时区名称 + 本地时间。
- 非推荐时段应写成”谨慎选项 / 受影响对象”，不要写成”不可用”。
