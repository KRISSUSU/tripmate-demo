# hi-calendar

技能核心能力通过 CLI 提供。命中本文件已披露参数的高频场景时可直接执行；参数不确定、命令失败、需求超出高频路径或本 skill 与 CLI 冲突时，必须查看 CLI help，**禁止猜测命令用法**：

```bash
# 查看具体命令的参数、示例和输出格式
hi calendar --help
hi calendar:<method> --help
```

# 环境准备

## 通用执行模型

所有日历任务都按同一套模型处理，不要从命令倒推流程：

1. **识别意图**：创建、推荐、查询、修改、取消、专注时间。
2. **收敛事实**：时间、人、空间资源、文档、循环范围、时区、权限。
3. **生成候选**：智能推荐、纯时间候选、纯会议室候选、日程候选。
4. **决策确认**：用户选择候选、确认冲突风险、确认循环范围和文档决策。
5. **执行写入**：只有用户明确确认最终计划后，才调用 create / edit / cancel 类命令。

查询命令只提供事实；候选命令只提供方案；写命令只在确认后执行。不要把候选结果说成已创建、已修改或已取消。

### 高频参数速查

以下只覆盖高频主路径；参数不确定、命令失败或遇到低频变体时，再查看对应命令 `--help`。

#### 人员搜索 / 参会人解析

```bash
hi search:employee --query "dot"
```

- `--query`：邮箱 / 员工姓名 / 薯名；

#### 时区确认

```bash
hi calendar:get-timezone --account-ids a@xiaohongshu.com,b@xiaohongshu.com
```

- `--account-ids`：多个账号用逗号分隔；不传则查当前用户。
- 关键输出：`timeZoneIana` 用于本地时间换算；`utcOffset` 用于展示偏移；`localNow` 用于解释“今天 / 明天”等相对时间。

## 会议创建的资源模型

创建会议不是机械补参数，而是编排资源：

- **时间资源**：用户给定、推荐产生，或由候选方案确认得到。
- **人员资源**：用户给定、搜索解析；同名或多候选时必须确认。
- **空间资源**：用户指定、明确不需要、由智能推荐产生，或由纯会议室分支产生。
- **材料资源**：文档链接、授权/任务决策，或明确无文档。
- **执行授权**：最终计划复述 + 用户明确确认。

单次会议默认使用**智能推荐**收敛时间和会议室：当时间、参会人已明确，且用户没有明确表示“不需要会议室 / 线上会议 / 无需会议室”时，使用 `calendar:get-intelligent-recommendation`。会议室可以是智能推荐后的候选，不要在智能推荐前强行追问具体会议室。

## 最佳实践

### `get-timezone` / 时区确认

- 因为有跨时区、多人会议场景，必须先获取**创建人和参会人**时区。
- 创建人时区缺失时停止询问；参会人时区缺失时可按创建人时区继续，但必须提示风险。

### 智能推荐会议

`calendar:get-intelligent-recommendation` 是单次会议的**重点推荐入口**，用于生成**时间+会议室**候选项。

- 适用：已明确开始时间、结束时间、参会人，且用户没有明确表示“不需要会议室 / 线上会议”。
- 不适用：循环会议、纯查共同空闲、明确无会议室、只有创建人无其他参会人。
- 展示：**必须保留时间和会议室两层信息**；同一时间下多个会议室时，让用户按候选编号选择时间和房间。
- 回流：用户选定候选后，回到写前确认；不要在推荐分支里直接创建。

## 全局护栏

- 关键事实不明确时停止并补齐；不要猜时间、参会人、会议室、区域、文档或取消范围。
- 所有返回的候选项**必须展示并让用户自行选择**；**不得静默选择第一个**。
- 智能推荐候选包含时间和会议室两层信息，必须一起展示给用户选择。
- 写操作前必须复述最终计划并获得明确确认。
- 创建或编辑会议的最终确认**必须包含文档决策**：具体文档或明确无文档。
- 查询失败按失败处理；不要把失败解释成无结果，不要换日期、换目标或用相似结果替代。
- 用户明确要求跳过检查时可以继续，但必须说明跳过了什么、风险是什么。
- 会议纪要定位只交接 `tencentMeetingDetail.meetingId`；字段缺失时停止说明。候选展示遵循 `references/hi-calendar-output-guidelines.md` 的最小披露规则。
